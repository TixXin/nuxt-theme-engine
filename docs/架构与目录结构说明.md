# `@tixxin/nuxt-theme-engine` 架构与目录结构说明

本文档面向希望深入了解模块内部机制的开源贡献者和架构设计者，说明源码目录、核心原理以及本地开发方式。

## 1. 模块目录结构

```text
nuxt-theme-engine/
├── src/
│   ├── module.ts
│   ├── runtime/
│   │   ├── components/
│   │   │   └── ThemeComponent.vue
│   │   ├── composables/
│   │   │   ├── useThemeEngine.ts
│   │   │   └── useThemeConfig.ts
│   │   └── plugins/
│   │       └── theme-sync.client.ts
│   └── utils/
│       ├── parseThemeJson.ts
│       ├── generateAlias.ts
│       ├── checkCssVars.ts
│       └── generateDts.ts
├── playground/
├── contracts/
├── package.json
└── build.config.ts
```

### 职责划分
- `src/module.ts`：模块构建时入口，负责扫描主题、注册钩子、注入模板与运行时代码。
- `src/runtime/`：运行时逻辑，包括主题分发组件、状态管理、配置读取和 DOM 同步。
- `src/utils/`：构建时工具，如解析 `theme.json`、生成继承别名、校验 CSS 变量、生成类型文件。
- `playground/`：本地调试用 Nuxt 应用。
- `contracts/`：组件契约定义，可独立为 `@tixxin/theme-contracts`。

## 2. 核心原理

### 2.1 构建时前缀化与继承别名
引擎在构建阶段扫描 `themesDir` 下所有主题：

1. 读取每个主题的 `theme.json`。
2. 建立继承树并检测循环依赖。
3. 为组件生成主题前缀名称，例如 `AuroraPostList`。
4. 为未覆盖的父主题组件生成 fallback 别名。

### 2.2 运行时动态分发
`<ThemeComponent>` 接收逻辑组件名，结合 `currentTheme` 计算目标组件名，并使用 `resolveComponent` 或异步映射表完成渲染。

### 2.3 按需加载静态映射
开启 `lazyLoadThemes` 后，构建时生成静态异步组件映射，避免把所有主题组件打进初始包。

### 2.4 CSS 变量校验
在构建完成后扫描最终 CSS 产物，对比 `requiredCssVars`，输出变量覆盖报告。

## 3. 高级扩展设计边界

- 核心业务数据走强类型契约。
- 主题展示配置走 `useThemeConfig()` 这样的 KV 抽象。
- 主题可以自定义布局和专属组件，但不应破坏主项目的业务主模型边界。

## 4. 本地开发与调试

### 4.1 环境准备

```bash
git clone https://github.com/tixxin/nuxt-theme-engine.git
cd nuxt-theme-engine
pnpm install
```

### 4.2 启动调试

```bash
pnpm dev
```

### 4.3 调试说明
- 修改 `src/runtime/` 下代码通常可直接 HMR。
- 修改 `src/module.ts` 或 `src/utils/` 下代码通常需要重启开发进程。
- 可检查 `.nuxt/` 目录中的生成产物验证类型和别名是否正确。

### 4.4 测试与构建

```bash
pnpm test:unit
pnpm test:e2e
pnpm build
```
