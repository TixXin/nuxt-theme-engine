# `@tixxin/nuxt-theme-engine` 架构与目录结构说明

本文档面向希望了解模块内部工作方式的贡献者，重点说明源码结构、核心机制与默认扩展边界。

## 1. 仓库结构

```text
nuxt-theme-engine/
├── src/
│   ├── module.ts
│   ├── runtime/
│   └── utils/
├── packages/
│   └── theme-contracts/
├── playground/
└── docs/
```

### 主要目录职责

- `src/module.ts`：模块入口，负责扫描主题、注册组件、注入构建模板和运行时代码
- `src/runtime/`：运行时逻辑，包括 `<ThemeComponent>`、`useThemeEngine()`、`useThemeConfig()` 和客户端同步插件
- `src/utils/`：构建时工具，如 `theme.json` 解析、继承链处理、别名生成、CSS 变量校验、类型声明生成
- `packages/theme-contracts/`：默认博客示例契约，不是引擎唯一可用契约
- `playground/`：示例应用，既用于主题演示，也用于回归验证默认契约和本地自定义契约
- `docs/`：公开长期文档

## 2. 核心机制

### 2.1 主题扫描与继承

引擎在构建阶段扫描 `themesDir`：

1. 读取每个主题的 `theme.json`
2. 校验主题名称与继承配置
3. 构建继承链并检测循环依赖
4. 收集主题组件和样式文件

### 2.2 构建时别名与 fallback

每个主题组件都会被映射成带前缀的唯一别名，例如：

- `AuroraPostList`
- `TixClassicHomeLayout`

如果子主题未实现某个契约组件，引擎会自动回退到继承链中的父主题实现。

### 2.3 运行时分发

页面层统一使用 `<ThemeComponent>`：

```vue
<ThemeComponent name="PostList" :posts="posts" />
```

运行时根据当前主题和构建生成的映射表，解析到具体主题实现。

### 2.4 类型生成

引擎会从契约入口中读取：

- `ThemeComponentContracts`
- `themeContractNames`

并据此生成 build-time 类型声明，为 `<ThemeComponent>` 的 `name` 和 Props 提供约束。

### 2.5 可替换契约入口

当前仓库内置 `packages/theme-contracts` 作为默认博客契约，但引擎支持通过以下配置切换到任意契约入口：

```ts
themeEngine: {
  contractsEntry: '#theme-contracts',
  contractsImportId: '#theme-contracts'
}
```

这使引擎可以用于博客、文档站、电商、社区等不同业务场景。

## 3. 设计边界

- 引擎负责主题扫描、分发、类型生成和样式治理，不负责业务路由定义
- 契约是业务相关的，可替换；引擎本体应保持业务中立
- 核心业务数据建议通过契约强类型传递
- 主题特有展示配置建议通过 `useThemeConfig()` 管理
- 主题可以拥有专属内部组件，但只有被提升为公共契约的组件才进入 `<ThemeComponent>` 分发体系

## 4. 本地开发

### 4.1 常用命令

```bash
pnpm install
pnpm dev
pnpm dev:prepare
pnpm typecheck
pnpm build
```

### 4.2 调试建议

- 修改 `src/runtime/` 下文件通常可直接 HMR
- 修改 `src/module.ts` 或 `src/utils/` 后，建议重新执行 `pnpm dev`
- 若需要检查生成结果，可查看 `playground/.nuxt/` 下的模板和类型文件
