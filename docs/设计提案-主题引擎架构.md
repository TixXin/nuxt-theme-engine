# 设计提案：主题引擎架构

## 背景

传统“换肤”方案通常只解决颜色和少量样式差异，但在真实内容站或博客系统中，不同主题往往还意味着：

- 不同的布局结构
- 不同的组件实现
- 不同的主题专属挂件
- 不同的主题展示配置

`@tixxin/nuxt-theme-engine` 的目标，是让这类差异可以被抽象成一套可复用的 Nuxt 4 主题引擎，而不是把主题逻辑散落在业务项目内部。

## 目标

- 支持多主题目录扫描与继承
- 支持运行时按主题分发组件
- 支持契约驱动的类型提示与 Props 校验
- 支持按主题隔离的展示配置
- 支持 CSS 变量治理与主题样式隔离
- 支持默认契约与自定义契约两种模式

## 关键设计

### 1. 主题即目录层

每个主题都包含：

- `theme.json`
- `app/components/`
- `app/assets/`

并可通过 `extends` 继承父主题。

### 2. 构建时注册，运行时分发

引擎在构建阶段：

1. 扫描所有主题
2. 建立继承链
3. 为组件生成带前缀的唯一别名
4. 为缺失组件生成 fallback 映射

页面层在运行时只依赖逻辑组件名：

```vue
<ThemeComponent name="PostList" :posts="posts" />
```

### 3. 契约驱动类型安全

引擎不直接绑定具体业务模型，而是从契约入口读取：

- `ThemeComponentContracts`
- `themeContractNames`

再自动生成 `<ThemeComponent>` 所需的类型声明。

默认仓库中提供 `packages/theme-contracts` 作为博客示例契约，但使用者也可以切换为：

- 自己的 npm 契约包
- 项目内本地契约文件

### 4. 主题配置与业务数据分层

- 核心业务数据：建议通过契约强类型传递
- 主题展示配置：通过 `useThemeConfig()` 做按主题隔离的 KV 存储

这样可以避免把主题系统变成业务逻辑收容层。

### 5. 样式治理

- 所有主题样式应作用于 `[data-theme="xxx"]`
- 主题变量统一采用 `--theme-*`
- 可使用 CSS、SCSS、UnoCSS、Tailwind 等任意样式方案
- 引擎可对必需 CSS 变量进行校验

## 取舍

### 为什么不是纯运行时动态解析

纯运行时查找虽然灵活，但会带来：

- 更高的运行时开销
- 更弱的类型提示
- 更复杂的 HMR 与调试体验

因此本项目选择“构建时映射 + 运行时轻分发”的方案。

### 为什么契约不写死在引擎里

因为契约天然带有业务属性：

- 博客需要 `PostList`
- 电商需要 `ProductCard`
- 文档站需要 `DocSidebar`

所以引擎应该通用，契约应该可替换。

## 当前默认实现

当前仓库提供：

- 默认博客示例契约：`packages/theme-contracts`
- 示例主题：`base`、`aurora`、`tix-three-column`、`tix-classic`
- `playground` 中的本地契约示例：`playground/theme-contracts`

这既方便演示，也方便验证“默认契约”和“自定义契约”两条路径。

## 未来可扩展方向

- 更完善的 DevTools 面板
- 更细粒度的主题懒加载策略
- 更强的契约生成与校验机制
- 面向不同业务域的官方示例契约包
