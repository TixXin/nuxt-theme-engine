# `@tixxin/nuxt-theme-engine` 用户使用指南

本指南说明如何在 Nuxt 4 项目中接入主题引擎，以及如何开发一个兼容该引擎的自定义主题。

## 1. 安装与初始化

```bash
pnpm add @tixxin/nuxt-theme-engine
```

默认博客契约已经随 `@tixxin/nuxt-theme-engine` 一起发布，安装主题引擎时不需要再单独安装 `@tixxin/theme-contracts`。

在 `nuxt.config.ts` 中配置：

```ts
export default defineNuxtConfig({
  modules: ['@tixxin/nuxt-theme-engine'],
  themeEngine: {
    themesDir: '~/themes',
    defaultTheme: 'nexus',
    lazyLoadThemes: true,
    cookieKey: 'my-site-theme',
    requiredCssVars: ['--theme-primary', '--theme-bg']
  }
})
```

## 2. 在项目中使用主题引擎

### 2.1 使用 `<ThemeComponent>`

```vue
<template>
  <!-- 基础用法：通过 default slot 传入内容 -->
  <ThemeComponent name="HomeLayout" :title="pageTitle">
    <ThemeComponent name="PostList" :posts="posts" />
  </ThemeComponent>
</template>
```

支持 **named slots**，用于多栏布局的精细分发：

```vue
<template>
  <ThemeComponent name="HomeLayout">
    <template #nav>
      <ThemeComponent name="SidebarNav" :items="navItems" current-path="/" />
    </template>
    <template #widgets>
      <ThemeComponent name="SubscribeCard" />
    </template>
    <template #default>
      <ThemeComponent name="PostList" :posts="posts" />
    </template>
    <template #aside>
      <ThemeComponent name="SiteStats" :stats="stats" />
    </template>
    <template #footer>
      <ThemeComponent name="BlogFooter" copyright="2026 My Blog" />
    </template>
  </ThemeComponent>
</template>
```

> 各主题的 HomeLayout 自行决定如何排列这些 slot；未使用的 slot 会被静默忽略。

### 2.2 使用 `useThemeEngine()`

```vue
<script setup lang="ts">
const { currentTheme, availableThemes, setTheme } = useThemeEngine()

function handleSwitch(themeName: string) {
  setTheme(themeName)
}
</script>
```

## 3. 如何开发一个新主题

### 3.1 目录结构

```text
themes/my-theme/
├── theme.json
└── app/
    ├── assets/
    │   ├── theme.css        ← CSS 变量
    │   ├── layout.css       ← 布局样式
    │   └── ...              ← 更多模块化 CSS
    └── components/
        ├── HomeLayout.vue
        ├── PostList.vue
        └── ...
```

主题引擎会自动扫描 `app/assets/` 下的 CSS 文件并注入，**不需要** 手动在 `nuxt.config.ts` 中配置。

### 3.2 `theme.json`

```json
{
  "name": "my-theme",
  "label": "My Theme",
  "description": "自定义主题描述",
  "extends": "base"
}
```

- `name`：必填，匹配 `^[a-z0-9-]+$`
- `label`：可选，用于 UI 显示
- `description`：可选
- `extends`：可选，指定父主题名称，未实现的组件自动回退到父主题

### 3.3 样式隔离规范

1. 所有主题全局样式必须包裹在 `[data-theme="主题名"]` 下。
2. 主题变量统一以 `--theme-` 开头。
3. Vue 组件中推荐 **不写 `<style>` 块**，将样式抽到 `app/assets/*.css` 模块化管理。

```css
[data-theme="my-theme"] {
  --theme-bg: #0d1117;
  --theme-text: #e6edf3;
  --theme-accent: #3fb950;
  --theme-muted: rgba(139, 148, 158, 0.7);
}
```

### 3.4 实现契约组件

主题引擎本身不强绑定某一套业务契约。默认情况下，它会直接使用包内置的博客示例契约，导入路径为 `@tixxin/nuxt-theme-engine/default-contracts`。仓库里的 `@tixxin/theme-contracts` 只是开发期/示例来源，你可以：

1. 直接复用默认契约
2. 发布你自己的契约包
3. 在项目内维护本地契约入口文件

如果复用默认契约，可以直接从 `@tixxin/nuxt-theme-engine/default-contracts` 导入 Props 接口：

```vue
<script setup lang="ts">
import type { PostListProps } from '@tixxin/nuxt-theme-engine/default-contracts'

defineProps<PostListProps>()
</script>

<template>
  <div class="my-post-list">
    <div v-for="post in posts" :key="post.id">{{ post.title }}</div>
  </div>
</template>
```

当前可用契约：`HomeLayout`、`PostList`、`PostDetail`、`SidebarNav`、`SiteStats`、`SubscribeCard`、`BlogFooter`。

#### 使用自定义契约入口

如果你的项目不是博客场景，推荐指定自定义契约入口：

```ts
export default defineNuxtConfig({
  alias: {
    '#theme-contracts': './theme-contracts/index.ts'
  },
  modules: ['@tixxin/nuxt-theme-engine'],
  themeEngine: {
    contractsEntry: '#theme-contracts',
    contractsImportId: '#theme-contracts'
  }
})
```

自定义契约入口至少应导出：

- `ThemeComponentContracts`
- `themeContractNames`

引擎会基于 `themeContractNames` 生成 `<ThemeComponent>` 的 `name` 联合类型，并基于 `ThemeComponentContracts` 生成 Props 分发类型。

### 3.5 使用 UnoCSS（可选）

如果你的主题需要使用 UnoCSS，在 playground 或宿主项目中：

```bash
pnpm add -D @unocss/nuxt unocss @unocss/transformer-directives
```

在 `nuxt.config.ts` 中注册，并在 `uno.config.ts` 中启用 `transformerDirectives` 以支持 CSS 文件中的 `@apply` 指令。

## 4. 高级扩展建议

- 如果只是展示层配置差异，优先使用 `useThemeConfig()` 管理主题特有配置。
- 如果你维护的是一条明确业务线（例如博客系统），可以维护自己的“默认契约包”并在多主题之间复用。
- 如果是跨主题通用数据扩展，优先升级你当前项目使用的契约入口，而不是把所有项目都绑定到同一个公共包。
- 如果是某个主题独有的小部件，可以直接在主题内部定义专属组件。

## 5. 本地开发

### 5.1 启动 playground

```bash
pnpm install
pnpm dev
```

默认会启动 `playground/` 下的 Nuxt 应用，用于验证：

- 主题切换是否生效
- `ThemeComponent` 是否正确分发
- 父主题 fallback 是否生效
- `useThemeConfig()` 的主题隔离是否正常

### 5.2 构建与类型检查

```bash
pnpm dev:prepare
pnpm typecheck
pnpm build
```

- `pnpm dev:prepare` 会生成 `playground/.nuxt` 类型产物。
- `pnpm typecheck` 会同时检查模块源码和 playground。
- `pnpm build` 会构建主题引擎模块及其内置默认契约；如果你切换为自定义契约包，则应先确保你的契约入口可被正常解析。
- 开发环境下即使配置了 `lazyLoadThemes: true`，模块也会默认退回到同步注册模式，以避免 HMR 与 hydration 问题；生产构建时才会启用真正的静态异步映射。

## 6. IDE 报错排查

如果编辑器里出现“找不到 `defineNuxtConfig` / `useThemeEngine` / `#build/*` / `@tixxin/nuxt-theme-engine/default-contracts`”之类报错，优先检查下面几项：

1. 先执行一次 `pnpm dev:prepare`，确保 `playground/.nuxt` 类型文件已经生成。
2. 确认 `playground/tsconfig.json` 存在，并继承 `playground/.nuxt/tsconfig.json`。
3. 确认 `src/runtime/tsconfig.json` 与 `src/runtime/build-shims.d.ts` 存在，这两者用于让模块运行时代码在编辑器中识别 `#build/theme-engine.*` 虚拟模块。
4. 如果依旧报红，重启 TypeScript Server 或重开编辑器窗口。

### 6.1 为什么模块源码和 playground 需要两套类型环境

- `playground/` 是一个真实的 Nuxt 应用，需要继承 `.nuxt` 生成的类型。
- `src/runtime/` 是模块源码，不直接位于 Nuxt 应用根目录，所以需要单独的 `tsconfig` 和 `#build` 声明文件来辅助 IDE 理解虚拟模块。
