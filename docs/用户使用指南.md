# `@tixxin/nuxt-theme-engine` 用户使用指南

本指南说明如何在 Nuxt 4 项目中接入主题引擎，以及如何开发一个兼容该引擎的自定义主题。

## 1. 安装与初始化

```bash
pnpm add @tixxin/nuxt-theme-engine
```

在 `nuxt.config.ts` 中配置：

```ts
export default defineNuxtConfig({
  modules: ['@tixxin/nuxt-theme-engine'],
  themeEngine: {
    themesDir: '~/themes',
    defaultTheme: 'nexus',
    lazyLoadThemes: true,
    cookieKey: 'my-site-theme',
    requiredCssVars: ['--theme-primary', '--theme-bg']
  }
})
```

## 2. 在项目中使用主题引擎

### 2.1 使用 `<ThemeComponent>`

```vue
<template>
  <ThemeComponent name="HomeLayout" :title="pageTitle">
    <template #content>
      <p>这是传入插槽的内容</p>
    </template>
  </ThemeComponent>
</template>
```

### 2.2 使用 `useThemeEngine()`

```vue
<script setup lang="ts">
const { currentTheme, availableThemes, setTheme } = useThemeEngine()

function handleSwitch(themeName: string) {
  setTheme(themeName)
}
</script>
```

## 3. 如何开发一个新主题

### 3.1 目录结构

```text
themes/aurora/
├── nuxt.config.ts
├── theme.json
└── app/
    ├── assets/css/theme.css
    └── components/
        └── HomeLayout.vue
```

### 3.2 `theme.json`

```json
{
  "name": "aurora",
  "displayName": "Aurora 极光",
  "description": "一个带动态顶栏的双栏主题",
  "version": "1.0.0",
  "extends": "base-theme",
  "previewColors": ["#f97316", "#ec4899"]
}
```

### 3.3 `nuxt.config.ts`

```ts
import { defineNuxtConfig } from 'nuxt/config'
import { fileURLToPath } from 'node:url'
import { dirname, join } from 'node:path'

const currentDir = dirname(fileURLToPath(import.meta.url))

export default defineNuxtConfig({
  css: [join(currentDir, './app/assets/css/theme.css')]
})
```

### 3.4 样式隔离规范

1. 所有主题全局样式必须包裹在 `[data-theme="主题名"]` 下。
2. 主题变量统一以 `--theme-` 开头。

```css
[data-theme="aurora"] {
  --theme-primary: #f97316;
  --theme-bg: #fafaf9;
  --theme-text: #292524;
}

.dark [data-theme="aurora"] {
  --theme-bg: #1c1917;
  --theme-text: #fafaf9;
}
```

### 3.5 实现契约组件

```vue
<script setup lang="ts">
defineProps<{
  title: string
}>()
</script>
```

## 4. 高级扩展建议

- 如果只是展示层配置差异，优先使用 `useThemeConfig()` 管理主题特有配置。
- 如果是跨主题通用数据扩展，优先升级 `@tixxin/theme-contracts`。
- 如果是某个主题独有的小部件，可以直接在主题内部定义专属组件。

## 5. 本地开发

### 5.1 启动 playground

```bash
pnpm install
pnpm dev
```

默认会启动 `playground/` 下的 Nuxt 应用，用于验证：

- 主题切换是否生效
- `ThemeComponent` 是否正确分发
- 父主题 fallback 是否生效
- `useThemeConfig()` 的主题隔离是否正常

### 5.2 构建与类型检查

```bash
pnpm dev:prepare
pnpm typecheck
pnpm build
```

- `pnpm dev:prepare` 会生成 `playground/.nuxt` 类型产物。
- `pnpm typecheck` 会同时检查模块源码和 playground。
- `pnpm build` 会先构建 `@tixxin/theme-contracts`，再构建主题引擎模块。
- 开发环境下即使配置了 `lazyLoadThemes: true`，模块也会默认退回到同步注册模式，以避免 HMR 与 hydration 问题；生产构建时才会启用真正的静态异步映射。

## 6. IDE 报错排查

如果编辑器里出现“找不到 `defineNuxtConfig` / `useThemeEngine` / `#build/*` / `@tixxin/theme-contracts`”之类报错，优先检查下面几项：

1. 先执行一次 `pnpm dev:prepare`，确保 `playground/.nuxt` 类型文件已经生成。
2. 确认 `playground/tsconfig.json` 存在，并继承 `playground/.nuxt/tsconfig.json`。
3. 确认 `src/runtime/tsconfig.json` 与 `src/runtime/build-shims.d.ts` 存在，这两者用于让模块运行时代码在编辑器中识别 `#build/theme-engine.*` 虚拟模块。
4. 如果依旧报红，重启 TypeScript Server 或重开编辑器窗口。

### 6.1 为什么模块源码和 playground 需要两套类型环境

- `playground/` 是一个真实的 Nuxt 应用，需要继承 `.nuxt` 生成的类型。
- `src/runtime/` 是模块源码，不直接位于 Nuxt 应用根目录，所以需要单独的 `tsconfig` 和 `#build` 声明文件来辅助 IDE 理解虚拟模块。
